## 1. 产品概述与目标用户

### 1.1 产品定位

Woo（无我笔记）是一款跨平台 Markdown 桌面笔记软件，核心定位是：

- **极致简洁的写作环境**，减少干扰，专注创作
- **融合思维导图大纲**，提供结构化思考能力
- **Git 版本管理**，实现安全可靠的数据同步
- **AI 辅助能力**，提升写作效率与质量

### 1.2 目标用户

- **内容创作者**：作家、博主、记者等需要长期专注写作的人群
- **知识工作者**：学生、研究人员、程序员等需要整理笔记和知识的人群
- **思维导图爱好者**：喜欢结构化思考和视觉化表达的用户
- **极简主义者**：追求简洁界面和高效工作流的用户

### 1.3 核心价值主张

1. **极简设计**：完全避免不必要的视觉干扰
2. **双视图协同**：Markdown 编辑与思维导图大纲无缝切换
3. **版本控制**：基于 Git 的可靠数据管理和同步
4. **AI 赋能**：智能写作辅助，提升创作效率

#### 2.5.4 技术实现要点

1. **组件自我递归**
   - `DropdownMenu` 组件内部调用自身实现无限级子菜单
   - 通过 `isSubmenu` prop 控制样式（普通菜单/子菜单定位）

2. **事件处理机制**
   ```typescript
   // 统一事件处理函数
   const handleMenuAction = (action: string) => {
     // 根据 action 类型执行不同操作
     switch(action) {
       case 'new-document':
         // 创建新文档逻辑
         break
       case 'import-markdown':
         // 导入 Markdown 逻辑
         break
       // ... 更多 action
     }
   }
   ```

3. **交互体验优化**
   - 点击外部区域自动关闭下拉菜单
   - 子菜单悬停展开（鼠标进入/离开）
   - 平滑过渡动画（opacity + transform）
   - 快捷键支持（待实现）

4. **扩展性设计**
   - 新增菜单只需在 `config/menus.ts` 添加配置
   - 支持动态菜单（根据权限/状态显示不同菜单项）
   - 支持图标菜单（可在 MenuItem 中扩展 icon 字段）

## 3. 技术架构建议

### 3.1 技术栈选择

**前端**：Vue 3 + TypeScript + Electron
**后端**：Java 17 + Spring Cloud
**存储**：Redis + MySQL
**部署**：Docker

**选择理由**：

1. **前端**：Vue3 + TypeScript 提供优秀的开发体验，Electron 确保跨平台支持
2. **后端**：Java 17 + Spring Cloud 提供稳定可靠的服务端架构
3. **存储**：MySQL 存储结构化数据，Redis 提升缓存性能
4. **部署**：Docker 简化部署流程，提高环境一致性

### 3.2 整体架构设计

- **框架**：Vue 3 + TypeScript

## 4. 数据模型与存储设计

### 4.1 本地存储结构

```
Woo Notes/
├── config/
│   ├── settings.json          # 应用设置
│   ├── github_token.json      # GitHub 授权信息（加密存储）
│   └── image_host.json        # 图床配置
├── data/
│   ├── notes.db               # SQLite 数据库（笔记元数据）
│   └── cache/                 # 缓存目录
└── repositories/              # Git 仓库目录
    └── [username]/
        └── [repo-name]/       # 每个 GitHub 仓库对应一个目录
            ├── note-1/        # 笔记 1（分支：note/note-1）
            └── note-2/        # 笔记 2（分支：note/note-2）
```

### 4.2 数据模型

#### 4.2.1 笔记模型 (Note)

```typescript
interface Note {
  id: string;                    // 唯一标识符（UUID）
  title: string;                 // 笔记标题
  content: string;               // Markdown 内容
  createdAt: Date;               // 创建时间
  updatedAt: Date;               // 更新时间
  repositoryId: string;          // 关联的仓库 ID
  branchName: string;            // Git 分支名
  tags: string[];                // 标签列表
  metadata: {
    wordCount: number;           // 字数统计
    lastSyncAt?: Date;           // 最后同步时间
    syncStatus: 'idle' | 'syncing' | 'conflict';
  };
}
```

#### 4.2.2 仓库模型 (Repository)

```typescript
interface Repository {
  id: string;
  owner: string;                 // GitHub 用户名
  name: string;                  // 仓库名
  localPath: string;             // 本地路径
  defaultBranch: string;         // 默认分支
  isPrivate: boolean;            // 是否私有
  lastSyncAt?: Date;
}
```

#### 4.2.3 配置模型 (Settings)

```typescript
interface Settings {
  editor: {
    fontFamily: string;
    fontSize: number;
    lineHeight: number;
    typewriterMode: boolean;
    typewriterPosition: number;  // 百分比，如 0.5 表示屏幕中央
    zoomLevel: number;           // 缩放级别，1.0 为默认
  };
  ui: {
    theme: 'light' | 'dark' | 'auto';
    layout: 'minimal' | 'three-column';
  };
  git: {
    autoSync: boolean;
    autoSyncInterval: number;    // 分钟
    commitMessageTemplate: string;
  };
  ai: {
    provider: 'openai' | 'claude' | 'custom';
    apiKey?: string;             // 加密存储
    model: string;
  };
}
```

### 4.3 Git 分支策略

**单笔记单分支模型**：

- 分支命名规则：`note/{note-id}`
- 每个笔记独立分支，互不干扰
- 主分支（main/master）用于仓库配置和全局文件

```
main
└── note/abc123 (笔记 A 的分支)
└── note/def456 (笔记 B 的分支)
```

### 4.4 冲突处理策略

1. **检测冲突**：pull 时检测是否有冲突
2. **冲突标识**：在编辑器中用 Git 标准格式标记冲突
3. **可视化解决**：提供三窗口对比（本地、远程、基础版本）
4. **手动解决**：用户编辑后标记为已解决
5. **自动备份**：解决前自动创建备份分支

## 5. 关键交互流程说明

### 5.1 笔记创建与编辑流程

```
用户点击新建笔记
    ↓
生成唯一 UUID
    ↓
创建本地笔记文件
    ↓
创建 Git 分支（note/{id}）
    ↓
初始化首次 commit
    ↓
进入编辑界面
    ↓
实时保存（防抖 2s）
    ↓
自动 commit（可选）
    ↓
手动/自动 push
```

### 5.2 GitHub 同步流程

```
用户授权 GitHub
    ↓
选择/创建仓库
    ↓
克隆仓库到本地
    ↓
拉取现有笔记分支
    ↓
[编辑笔记]
    ↓
用户点击同步
    ↓
commit 本地变更
    ↓
pull 远程变更
    ├─ 无冲突 → push
    └─ 有冲突 → 显示冲突解决界面
                  ↓
             用户解决冲突
                  ↓
             commit 解决结果
                  ↓
                  push
```

### 5.3 思维导图同步流程

```
用户编辑 Markdown 列表
    ↓
解析器分析列表结构
    ↓
生成思维导图数据结构
    ↓
更新导图视图
    ↓
[用户在导图中拖拽节点]
    ↓
更新数据结构
    ↓
反向生成 Markdown 列表
    ↓
更新编辑器内容
```

- **启动时间**：< 3 秒
- **文件打开**：< 1 秒（10MB 以内）
- **内存占用**：< 200MB（空闲状态）
- **同步延迟**：< 5 秒（网络正常情况下）

### 7.2 安全性要求

- **敏感数据加密**：GitHub Token、API Key 等使用系统密钥加密存储
- **HTTPS 传输**：所有网络请求使用 HTTPS
- **权限最小化**：只请求必要的 GitHub 权限（repo  scope）
- **本地数据隔离**：用户数据存储在用户目录下，权限受限

### 7.3 可靠性要求

- **自动备份**：每 10 分钟自动创建本地备份
- **数据完整性**：Git 保证版本历史可追溯
- **异常恢复**：崩溃后自动恢复未保存的内容
- **离线可用**：无网络时可正常编辑

### 7.4 可扩展性要求

- **插件化架构**：AI、图床等功能通过插件实现
- **模块化设计**：核心与扩展功能分离
- **API 设计**：预留插件开发接口

### 7.5 设计约束

- **极简原则**：UI 元素最少化，隐藏非必要功能
- **Effie 风格**：参考 Effie 的视觉语言和交互模式
- **性能优先**：避免不必要的动画和特效
- **可访问性**：支持键盘操作、屏幕阅读器（未来版本）

## 8. 风险与应对

| 风险            | 影响 | 概率 | 应对措施                        |
| ------------- | -- | -- | --------------------------- |
| Electron 包体积大 | 中  | 高  | 采用 ASAR 打包、代码分割、按需加载        |

| 跨平台适配难度       | 高  | 中  | V1.0 先聚焦 Windows，后续逐步扩展     |
| 后端性能瓶颈        | 中  | 低  | 采用 Redis 缓存、异步处理、MySQL 优化   |
